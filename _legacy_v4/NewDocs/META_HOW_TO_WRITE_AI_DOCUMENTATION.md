# META: Как да пишеш AI-focused документация

**Цел:** Документация която е само за Claude, без излишна информация, максимално практична.

---

## Принципи

### 1. Целева аудитория
- **Само за AI** (Claude Code)
- Предполага се техническо разбиране
- Без обяснения на основни концепции

### 2. Съдържание
**Включвай:**
- ✅ Конкретни файлове с пълни пътища
- ✅ Code snippets (минимални, работещи)
- ✅ Стъпки в последователност
- ✅ Критични детайли (напр. "requiredCommand ТРЯБВА да съвпада с commandType")
- ✅ Често срещани грешки и решения
- ✅ Проверъчен списък

**Изключвай:**
- ❌ Обяснения защо нещо работи по даден начин
- ❌ История на промени
- ❌ Теоретични обяснения
- ❌ Контекст който не е критичен за задачата
- ❌ Множество алтернативни подходи
- ❌ Философия на дизайна

### 3. Структура

```markdown
# Заглавие (ясна цел)

**Цел:** Какъв резултат се постига

---

## Стъпка 1: [Действие]

**Локация:** `/точен/път/до/файл`

**Действие:** Какво точно да направиш

[Code snippet ако е нужен]

**Важно:** Критична бележка

---

## Стъпка 2: ...

[повтаряй]

---

## Проверка

- ✅ Checklist item 1
- ✅ Checklist item 2

---

## Тестване

Как да провериш че работи

---

## Често срещани грешки

❌ **Проблем**
→ Решение
```

### 4. Стил

- **Кратко:** Максимум 2 изречения на параграф
- **Ясно:** Конкретни инструкции, не общи насоки
- **Практично:** Всичко трябва да е приложимо директно

### 5. Code Snippets

**Правила:**
- Минимални (само критичното)
- Работещи (не псевдокод)
- С коментари само за критични моменти
- Показват структура, не пълна имплементация

**Пример (добър):**
```typescript
{
  name: 'COMMAND_NAME',
  displayName: 'Name',
  isActive: true
}
```

**Пример (лош - много подробности):**
```typescript
// This object represents a command configuration
// that will be stored in the database and used
// by the generator service to create Arduino code
{
  name: 'COMMAND_NAME', // Must be uppercase
  displayName: 'Name', // User-friendly name
  description: 'Long description...', // Optional
  isActive: true // Set to false to disable
}
```

### 6. Naming Convention

**Формат:** `[TYPE]_[TOPIC].md`

**Types:**
- `HOW_TO_` - Стъпка по стъпка инструкции
- `REFERENCE_` - Справка (списъци, таблици)
- `TROUBLESHOOTING_` - Грешки и решения
- `META_` - Мета документация (като този файл)

**Примери:**
- `HOW_TO_ADD_NEW_ARDUINO_COMMAND.md`
- `REFERENCE_ARDUINO_COMMANDS.md`
- `TROUBLESHOOTING_SERIAL_CONNECTION.md`
- `META_HOW_TO_WRITE_AI_DOCUMENTATION.md`

### 7. Локация

**Папка:** `/Docs/NewDocs/`

**Организация:**
- Плоска структура (без подпапки)
- Имената на файловете са описателни
- Групирай с prefix (`HOW_TO_`, `REFERENCE_`, и т.н.)

## Когато създаваш нова документация

1. **Питай себе си:**
   - Каква е точната цел?
   - Какво минимално трябва да знам за да я постигна?
   - Кои файлове засягам?

2. **Избери тип:**
   - HOW_TO - ако има стъпки
   - REFERENCE - ако е списък/таблица
   - TROUBLESHOOTING - ако е за грешки
   - META - ако е за самата документация

3. **Пиши:**
   - Започни с "Цел:"
   - Стъпки със заглавия
   - Code snippets само критичното
   - Проверка и тестване
   - Често срещани грешки

4. **Ревю:**
   - Може ли да се премахне нещо без да загуби смисъл? → Премахни го
   - Всичко ли е конкретно и приложимо? → Ако не, конкретизирай
   - Има ли излишни обяснения? → Премахни ги

## Примери

### Добър пример
```markdown
## Стъпка 1: Добави команда

**Локация:** `/backend/src/data/seedCommands.ts`

**Действие:** Добави в `commandSeeds` масив:

\`\`\`typescript
{
  name: 'PULSE_COUNT',
  displayName: 'Pulse Count',
  isActive: true
}
\`\`\`

**Важно:** Restart backend след промяна.
```

### Лош пример
```markdown
## Стъпка 1: Добавяне на нова команда в seed файла

Командите в системата се зареждат от специален seed файл, който се намира
в backend частта на приложението. Този файл съдържа масив от команди които
се записват в MongoDB базата данни при стартиране на сървъра.

За да добавиш нова команда, трябва да отвориш файла, намираш се в папката
backend, след това src, след това data, и накрая файла seedCommands.ts...
```

## Ключови правила

1. **Цел винаги в началото**
2. **Конкретни пътища, не описания**
3. **Минимални code snippets**
4. **Само критична информация**
5. **Проверка и грешки накрая**
6. **Ако може да се премахне → премахни го**
